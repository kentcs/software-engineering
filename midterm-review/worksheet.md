# Software Engineering Midterm Practice Worksheet (Scenario Review)

This worksheet focuses on reasoning about software engineering decisions.  
Answer clearly and concisely.

--------------------------------------------------

## 1. Evidence and Testing

A team says:

"We tested the system and everything passed, so we know the software is correct."

1. What is wrong with this statement?

2. What does testing actually provide?

3. Why is testing still extremely valuable even though it cannot prove correctness?

--------------------------------------------------

## 2. Methodology Selection

Imagine three different projects.

Project A  
A startup is building a mobile app for restaurant reservations. The team expects to change features frequently based on customer feedback.

Project B  
A spacecraft control system must operate reliably for 10 years after launch, and the software cannot be modified once deployed.

Project C  
A small internal tool will be used by a team of analysts. The requirements are unclear and will likely evolve during development.

For each project:

1. What development methodology would you prefer?

2. Explain why that methodology fits the project.

--------------------------------------------------

## 3. Non-Agile Methodologies

Consider a traditional methodology that requires most requirements and architecture to be defined before implementation begins.

1. Why might this approach work well for some projects?

2. Why does this approach often fail in modern software environments?

3. Why do agile approaches attempt to avoid Big Design Up Front?

--------------------------------------------------

## 4. Spiral vs Agile Iterations

Two teams describe their development cycles.

Team A  
Each cycle focuses on identifying risks, evaluating solutions, and refining the design.

Team B  
Each cycle produces working software that users can test.

1. Which model resembles the spiral model?

2. Which resembles agile iteration?

3. What is the main purpose of each type of cycle?

--------------------------------------------------

## 5. Customer Communication

A development team builds a system entirely using unit tests written by developers.

After release, the customer says the system does not behave as expected.

1. Why might this happen even if all unit tests pass?

2. How could behavior-driven development reduce this risk?

3. Why are behavioral specifications often easier for customers to understand?

--------------------------------------------------

## 6. Behavioral Testing

A product owner helps write tests like this:

Given a user with a valid account  
When the user logs in with the correct password  
Then the system grants access

1. What development approach uses tests written in this style?

2. Why does this style help communication between developers and stakeholders?

3. How does this differ from traditional unit tests?

--------------------------------------------------

## 7. TDD Workflow

1. What is the first step in TDD?

3. What tells the developer that the next step is required?

4. When should refactoring occur?

5. Why does TDD encourage small incremental development?
